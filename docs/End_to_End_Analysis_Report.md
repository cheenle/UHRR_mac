## 3. 音频 TX 路径深度分析

### 3.1 数据流

```
麦克风输入
    ↓
Web Audio API (AudioContext)
    ↓ 采样率: 16kHz
    ↓ 格式: Float32
TX 均衡器 (可选)
    ↓ 低频/中频/高频调节
Int16 编码
    ↓ 格式: Int16 (50%带宽减少)
WebSocket (WSaudioTX)
    ↓ 二进制传输
服务器 WS_AudioTXHandler
    ↓ 解码
PyAudioPlayback
    ↓ 播放
音频输出设备 → 电台
```

### 3.2 关键参数

| 环节 | 参数 | 当前值 | 延迟估算 |
|------|------|--------|----------|
| 浏览器采样 | 采样率 | 16kHz | ~5ms |
| 音频缓冲 | frames_per_buffer | 256帧 | ~16ms |
| 网络传输 | WebSocket | 二进制 | ~5-20ms |
| 服务器缓冲 | PyAudio | 512帧 | ~32ms |
| **总计** | - | - | **~60-75ms** |

## 4. 音频 RX 路径深度分析

### 4.1 数据流

```
电台音频输入
    ↓
PyAudioCapture (独立线程)
    ↓ 采样率: 16kHz
    ↓ 格式: Float32 → Int16
    ↓ 缓冲: 256帧 (~16ms)
WebSocket (WSaudioRX)
    ↓ 二进制传输 (Int16)
    ↓ 带宽: ~256kbps
浏览器 WS_AudioRXHandler
    ↓ Int16 → Float32 解码
AudioWorklet (rx-player)
    ↓ 缓冲: 16-32帧
    ↓ 延迟: ~2-20ms
扬声器输出
```

### 4.2 关键参数

| 环节 | 参数 | 当前值 | 延迟估算 |
|------|------|--------|----------|
| PyAudio采集 | frames_per_buffer | 256帧 | ~16ms |
| 网络传输 | WebSocket | 二进制 | ~5-20ms |
| AudioWorklet | 缓冲区 | 16-32帧 | ~2-20ms |
| **总计** | - | - | **~25-60ms** |

## 5. PTT 控制路径深度分析

### 5.1 控制流程

```
用户按下PTT按钮
    ↓
tx_button_optimized.js: TXControl('start')
    ↓
├── 1. sendTRXptt(true) → 控制WebSocket
│       ↓ "setPTT:true"
│       ↓ 服务器处理
│       ↓ rigctld → 电台PTT=1
│
├── 2. toggleRecord(true) → 音频TX WebSocket
│       ↓ "m:16000,0,0,0"
│       ↓ 服务器初始化PyAudio
│
└── 3. 发送预热帧 → 确保音频数据到达
```

### 5.2 PTT 延迟分析

| 操作 | 预期延迟 | 备注 |
|------|----------|------|
| PTT命令发送 | ~5ms | WebSocket文本消息 |
| 服务器处理 | ~5-10ms | rigctld命令执行 |
| 音频初始化 | ~20-50ms | PyAudio设备初始化 |
| 预热帧发送 | ~10ms | 3帧×3ms间隔 |
| **总计** | **~40-75ms** | 从按钮按下到电台发射 |

## 6. ATR-1000 集成分析

### 6.1 数据流

```
ATR-1000 设备
    ↓ WebSocket (192.168.1.63:60001)
独立代理 (atr1000_proxy.py)
    ↓ Unix Socket (/tmp/atr1000_proxy.sock)
MRRC 桥接器
    ↓ 批量广播 (50ms批次)
WebSocket (/WSATR1000)
    ↓ 实时数据
移动端显示
    ↓ 功率/SWR
```

### 6.2 关键优化

| 优化项 | 实现 | 效果 |
|--------|------|------|
| 批量广播 | 50ms 批次收集 | 减少消息积压 |
| 线程安全 | IOLoop.add_callback | 主线程 WebSocket |
| 双重时间保护 | 500ms 最小间隔 | 防止压垮设备 |
| 连接预热 | 页面加载预连接 | PTT 响应 <200ms |

## 7. WebSocket 通信效率分析

### 7.1 连接架构

```
浏览器
├── wsControlTRX (控制通道)
│   └── 文本消息: setFreq, setMode, setPTT, PING/PONG
│
├── wsAudioTX (音频发送)
│   └── 二进制: Int16音频数据
│   └── 文本: "m:..."初始化, "s:"停止
│
├── wsAudioRX (音频接收)
│   └── 二进制: Int16音频数据
│
└── wsATR1000 (ATR-1000监控)
    └── JSON: 功率、SWR数据
```

### 7.2 消息频率分析

| 通道 | 消息类型 | 频率 | 方向 |
|------|----------|------|------|
| 控制通道 | 心跳PING | 5秒1次 | 双向 |
| 控制通道 | 状态查询 | 500ms1次 | 服务器→客户端 |
| 音频TX | 音频数据 | ~16ms1帧 | 客户端→服务器 |
| 音频RX | 音频数据 | ~16ms1帧 | 服务器→客户端 |
| ATR-1000 | SYNC | 500ms/2s | 按需 |

## 10. 结论

### 10.1 当前系统状态

| 指标 | 当前值 | 目标值 | 状态 |
|------|--------|--------|------|
| TX延迟 | ~65ms | <100ms | ✅ 达标 |
| RX延迟 | ~51ms | <100ms | ✅ 达标 |
| TX→RX切换 | ~60ms | <100ms | ✅ 达标 |
| 功率显示 | ~150ms | <200ms | ✅ 达标 |
| 带宽占用 | ~530kbps | <1Mbps | ✅ 达标 |
| PTT可靠性 | 99%+ | 99%+ | ✅ 达标 |

### 10.2 核心发现

1. **TX→RX切换延迟已大幅优化**: 从原来的 2-3 秒降低到 ~60ms
2. **ATR-1000 实时显示稳定**: 批量广播 + 双重时间保护机制有效
3. **音频延迟达到目标**: 端到端 ~65ms，满足实时通信需求
4. **带宽使用合理**: ~530kbps，适合大多数网络环境

### 10.3 系统稳定性

- PTT 可靠性: 99%+
- ATR-1000 稳定性: 稳定运行，无压垮风险
- 音频质量: 稳定，无明显抖动
- 内存管理: 自动清理，无泄露

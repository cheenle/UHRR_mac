# 阿拉丁 V2 业务工具包

> 面向通用软件团队的实操层。本文档不复述方法论叙事,只提供可直接复制使用的模板、清单和指标。
> 它把 `Aladdin_V2_Critical_Review.md` 中"可以直接拿走的五件事"工程化,补齐原方法论缺失的团队维度、纠错回路、成本/安全约束和硬指标。
>
> **定位**:批判分析的结论是——阿拉丁 V2 不是"不成熟",而是"成熟度高度不均衡":在单人硬件项目锤炼过的维度(执行约束、知识沉淀、架构记忆)很精良,在该项目从未施压的维度(多人判断、权威纠错、预算安全)接近空白。本工具包就是"把方法论开进新水域时要自己装的那套引擎"——补的不是坏零件,是射程外的新装备。

---

## 0. 怎么用这份文档

这不是又一份要通读的方法论。它是五个独立工具,按需取用:

| 工具 | 解决的问题 | 什么时候用 | 产出 |
|------|-----------|-----------|------|
| 1. 边界卡 | agent/人动手前没说清不做什么 | 每个任务开始前 | 10 行以内的一张卡 |
| 2. 七类 Harness | agent 缺乏安全轨道 | 项目建立时 + 每个 cycle 补充 | 可执行资产清单 |
| 3. 三个硬指标 | 杠杆只是口号,无法证伪 | 每个 release 后采样 | 三条趋势线 |
| 4. 纠错回路 | 老兵判断本身错了没人拦 | 每个关键决策点 | 一次轻量挑战 |
| 5. 失效场景标注 | 在错误场景里硬套方法 | 引入方法论时一次性 | 适用/不适用边界 |

最低门槛用法:只用工具 1(边界卡)。它 ROI 最高、迁移成本最低,今天就能用。其余四个随团队成熟度逐步引入。

## 工具 1:边界卡 (Boundary Card)

**原则:边界必须先于速度被写下来。没有边界的 vibe coding 只是高速猜测。**

在让 agent 或任何人动手前,强制产出这张卡。超过 10 行说明任务还没想清楚,不该开始。

```markdown
# 边界卡:[任务名]

- 目标(一句话):
- 不做什么(明确排除):
- 必须守住的(碰了就是事故):
- 最容易猜错的 3 点:
  1.
  2.
  3.
- 失败如何被发现(验证方式):
```

### 填写要点

- **"不做什么"比"做什么"更重要**。agent 的发散能力会自动扩张范围,边界卡的核心作用是收口。
- **"最容易猜错的 3 点"是老兵判断的显性化出口**。把"我知道但容易忘"的隐性经验,变成"下一个人能读到"的约束。写不出 3 点,说明对任务的风险理解还不够。
- **"失败如何被发现"决定这张卡是否可信**。如果失败只能靠人肉发现,这个任务不适合交给 agent。

### 反面信号

- 边界卡写完 5 分钟就被突破 → 目标本身定义错了,回到 Echo。
- "必须守住的"一栏为空 → 要么任务真的零风险(罕见),要么没想清楚(常见)。

## 工具 2:七类 Harness

**原则:Harness 是 agent 安全工作的轨道。没有 harness,工程问题会退化成"生成代码问题"。**

原方法论有六类。业务场景必须补第七类——这是从"个人项目"走向"业务系统"最大的缺口。

| # | Harness | 回答的问题 | 典型载体 | 业务关键点 |
|---|---------|-----------|----------|-----------|
| 1 | Context | 入口/边界/已知坑在哪 | `AGENTS.md`、架构索引 | 防止 agent 重新探索基础事实 |
| 2 | Spec | 这次要做什么、不做什么 | spec 文档、边界卡 | 范围先于实现 |
| 3 | Plan | agent 如何逐步执行 | checkbox 计划 | 每步可独立验证 |
| 4 | Verification | 怎么证明做对了 | 测试脚本、检查命令 | 连到可执行命令,不靠"看起来像" |
| 5 | Runtime | 怎么启动/停止/部署/看日志 | 运行脚本、Compose | 可复现的运行环境 |
| 6 | Memory | 同类问题怎么不再踩 | postmortem、ADR、决策记录 | 事故经验可复用 |
| **7** | **Safety/Cost** | **agent 能碰什么、花多少、谁批** | **预算上限、操作白名单、人工确认点** | **业务场景的硬约束** |

### 第七类 Harness 模板(新增,重点)

原方法论完全没有覆盖。在 AI 产能过剩时代,这是真正的业务风险来源。

```markdown
# Safety/Cost Harness:[项目/agent 名]

## 能力边界(agent 能碰什么)
- 允许:
  - [例:读写 feature 分支、运行测试、生成文档]
- 禁止(碰了立即停止):
  - [例:推送到 main/master、修改生产配置、删除数据、改动认证/权限代码]
- 需人工确认后才能执行:
  - [例:安装新依赖、修改 CI/CD、对外网络请求、删除多个文件]

## 成本边界
- 单任务 token/调用预算上限:
- 超预算时的行为:[停止并上报 / 降级 / 询问]
- 谁有权批准超预算:

## 安全红线
- 密钥/凭证:agent 不得读取的文件清单:
- 外发数据:不得向第三方端点传输的内容:
- 依赖供应链:新依赖必须 [固定版本 / 人工审核名称防 typosquatting]

## 生产环境人工确认点
- 触发条件:
- 确认人:
- 回滚方案:
```

### 引入策略

- 项目建立时一次性建好 Context / Runtime / Safety-Cost 三类(它们是地基)。
- 每个 cycle 增量补 Spec / Plan / Verification(它们随任务长出来)。
- Memory harness 在每次事故/复盘后强制更新——这是未来速度的本金。

## 工具 3:三个硬指标

**原则:衡量一次交付的终极标准是它是否让下一次更便宜。但"更便宜"必须能被证伪。**

原方法论的八个指标全是定性的,团队可以自称"健康"而无外部依据。用下面三个可量化的代理指标替换,能画出趋势线才算数。

| 硬指标 | 它替代的定性指标 | 怎么采集 | 健康趋势 |
|--------|-----------------|----------|----------|
| **接手探索时间** | "Harness 完整度" | agent/新人第一次改某模块前,从开始到能安全动手的耗时 | 随 harness 完善而下降 |
| **同类事故复发次数** | "记忆保真度" | 给事故打标签,统计同标签复发次数 | 趋近于 0;复发=Memory harness 失效 |
| **相似功能边际成本** | "Product 杠杆" | 第 N 次做同类功能的代码行数/工时 | 逐次下降;不降=没有真正抽象 |

### 采集模板

```markdown
# 杠杆度量:[版本/季度]

## 接手探索时间
- 模块:        | 本次:    分钟 | 上次:    分钟 | 趋势:↓/→/↑
- 改善动作:[补了什么 harness]

## 同类事故复发
- 事故标签:    | 历史发生:  次 | 本期复发:  次
- 根因是否已进 Memory harness:是/否

## 相似功能边际成本
- 功能类型:    | 第几次:  | 工时/行数:  | 对比首次:  %
- 复用了什么抽象:
```

### 关键判断

- 三条线里只要有一条**持续不降**,对应那层就是空转。
- 不要追求全测。先测"相似功能边际成本",它最直接反映方法论是否真的产生杠杆。

## 工具 4:纠错回路 (Correction Loop)

**原则:老兵判断是系统的最高权威,但当它本身错了的时候,原方法论没有纠错回路。**

这是原方法论最大的结构性缺陷。它假设老兵判断可靠,且由老兵自己决定"哪些现场事实值得采信"——这是一个潜在的闭环偏见。补一道轻量挑战机制,才敢用于团队。

在每个**关键决策点**(写进 ADR / guardrail 级别的判断)插入下面任一即可,不要求全做:

```markdown
# 纠错检查(三选一,关键决策点必做一次)

## 选项 A:同伴反向论证(成本最低)
- 一个人专门论证"这个判断为什么可能是错的"
- 至少给出 1 个具体失败场景
- 老兵需对该场景作出回应,而非忽略

## 选项 B:"如果老兵错了"压力测试
- 假设这个判断 100% 错误,会怎样?
- 错误多久后会被现场事实暴露?(反馈延迟)
- 暴露成本有多大?可逆吗?
- 若不可逆或暴露慢 → 必须做选项 A

## 选项 C:证据来源审计(针对"现场事实")
- 这条"现场事实"的来源是实测还是印象?
- 是否存在与之矛盾的其他证据被忽略?
- 采信它的人和定义问题的人是否同一个?(同一个=高风险)
```

### 何时必须触发(不可省略)

- 决策不可逆 或 错误暴露周期长(反馈延迟大)。
- 团队里只有一个判断来源(单点风险最高时,最需要外部挑战)。
- 强监管/合规/资金/数据安全相关的判断。

### 设计意图

成熟工程文化(设计评审、working backwards 等)都有刻意设计的"挑战权威"机制。这道回路不是不信任老兵,而是承认**任何单一判断源都需要一个证伪入口**——这恰好是方法论自己教的道理("现场事实优先于猜测"),只是要对老兵自己也适用。

## 工具 5:失效场景标注

**原则:一个标注了边界的方法论,才是成熟的方法论。**

阿拉丁 V2 的全部证据来自单一项目(领域稳定、判断集中、反馈周期长)。把它用到不匹配的场景会出问题——不是方法失败,是场景不同。引入方法论时,先做一次标注。

### 适用性矩阵

| 场景维度 | 高适用(方法直接用) | 低适用(需大幅改写) |
|---------|---------------------|---------------------|
| 领域稳定性 | 协议/规则多年不变(基础设施、硬件、平台) | 需求每周变(增长型 C 端、早期探索) |
| 判断集中度 | 有公认的技术负责人 | 判断分散、无公认老兵(新团队/新业务线) |
| 反馈周期 | 长(可承受 3-5 轮 Delta) | 短(需要小时级反馈) |
| 约束来源 | 现场实测高于文档 | **法规/合规高于现场**(金融、医疗、隐私) |
| 证据积累 | 有长期运行数据 | 全新系统,无历史数据 |

### 必须改写的两处

1. **FDE 节奏假设**:在需求高频变化的场景,"Delta 3-5 轮收敛"不成立。改为缩短单轮周期(小时级)、并把"反馈延迟"作为一等公民设计——agent 跑得越快,错误反馈延迟越致命。

2. **"现场优先于手册"原则**:在强监管领域**反转为"手册(法规)高于现场"**。违反会直接出事。这是原方法论的隐性前提(它来自硬件逆向),业务团队必须显式校正。

### 标注模板

```markdown
# 本团队的方法论适用边界

- 我们的领域稳定性:高 / 中 / 低 →
- 我们的判断集中度:集中 / 分散 →
- 我们能承受的反馈周期:
- 约束主来源:现场实测 / 法规手册 →
- 因此我们对原方法的改写:
  - FDE 节奏:
  - 现场 vs 手册优先级:
  - 其他:
```

# FDE 实践指导书

> **Forward Deployed Engineering — 从理论到实践的完整操作指南**
>
> 基于 MRRC 项目 (208 commits, V1.0→V5.2.0, 2024-10→2026-05) 的真实过程
> 结合 IBM TeamSD v2.3.2 (SDD 14 章) 方法论
> 面向个人开发者和小团队 (1-3 人)

---

## 导言：这份指导书适合谁？

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 个人开发者 / Indie Hacker | ★★★★★ | MRRC 即为 1 人项目，全程覆盖 |
| 2-5 人小团队 | ★★★★☆ | 角色轮转机制可直接套用 |
| 驻场工程师 (Forward Deployed) | ★★★★★ | 核心方法论即为 FDE |
| 远程/分布式团队 | ★★★☆☆ | Echo 阶段的驻场假设需调整 |
| 大中型产品团队 | ★★☆☆☆ | 部分模式适用，但组织复杂度超出范围 |

**你需要具备的前提**：
- 能独立完成全栈开发 (或至少有快速学习能力)
- 愿意在用户现场/真实场景中工作
- 接受"代码可以粗糙，成果必须可衡量"的理念

## 第一部分：方法论基础

### 1.1 FDE 的三个角色

```
┌──────────────────────────────────────────────────────────────────┐
│                        FDE 三角色模型                             │
│                                                                  │
│   Echo (需求发现)          Delta (原型交付)       Product (抽象泛化) │
│   ─────────────           ──────────────        ────────────────  │
│   "用户真正需要什么？"      "最快能跑是什么？"      "如何服务下10个客户？" │
│                                                                  │
│   嵌入式分析师              部署工程师             总部产品/工程     │
│   • 深入用户场景            • 快速原型              • 抽象通用模块    │
│   • 定义有价值的 demo       • 在时间线内交付        • 将碎石路修成    │
│   • 客户关系管理            • 代码粗糙但能跑         高速公路       │
│                                                                  │
│   典型产出:                典型产出:              典型产出:        │
│   • 问题陈述               • 可运行的原型          • 通用库/框架    │
│   • Demo 定义              • 快速修复             • 架构文档       │
│   • 用户反馈               • 功能验证             • 设计系统       │
│                                                                  │
│   时间占比: ~20%           时间占比: ~55%          时间占比: ~25%   │
└──────────────────────────────────────────────────────────────────┘
```

**关键认知**：
- 在同一个人切换角色的模式下，**最难的是角色切换**，不是技术本身
- Echo→Delta 切换需要"收窄聚焦"：从发散的问题发现转向收敛的交付
- Delta→Product 切换需要"抽离"：从具体代码中抬起头看模式

### 1.2 FDE 与 SDD (IBM TeamSD) 的对应关系

IBM TeamSD 的 14 章 SDD 为 FDE 提供了结构化框架。下面是每个 FDE 阶段应该驱动的 SDD 章节：

```
FDE 阶段         驱动哪些 SDD 章节                时机
────────        ────────────────                ────
Echo (需求发现)
  ├── 用户场景    → Ch4 系统上下文               首次 Echo
  ├── 业务目标    → Ch2 业务方向 (BUS 411)       首次 Echo
  ├── 需求识别    → Ch5 非功能需求 (ART 0507)    每次 Cycle
  ├── 用例定义    → Ch6 用例模型 (ART 0508)      每次 Cycle
  └── 可行性      → Ch13 可行性评估 (ART 0530)   首次 Echo

Delta (原型交付)
  ├── 领域建模    → Ch7 主题域模型 (APP 408)     每次 Cycle
  ├── 组件设计    → Ch11 组件模型 (ART 0515)     Cycle 中期
  ├── 服务定义    → Ch10 服务模型 (ART 0582)     Cycle 后期
  └── 部署实操    → Ch12 运维模型 (ART 0522)     Cycle 后期

Product (抽象泛化)
  ├── 架构决策    → Ch8 架构决策 (ART 0513)      抽象时记录
  ├── 架构全景    → Ch9 架构概览 (ART 0512)      每个版本
  ├── 项目范围    → Ch3 项目定义 (ENG 343)       每个版本
  ├── 执行摘要    → Ch1 执行摘要                 每个版本
  └── 版本历史    → Ch14 版本历史                每次发布
```

**核心原则**：**不要让文档驱动开发，让开发驱动文档**。每个 FDE 阶段完成后，更新对应的 SDD 章节。不要反过来。

### 1.3 FDE Cycle 的标准节奏

MRRC 经验数据 (基于 203 commits, 9 个 Cycle)：

| Cycle 类型 | 时长 | 版本跨度 | 文档产出 | 适用场景 |
|-----------|------|---------|---------|---------|
| **探索 Cycle** | 2-4 周 | 0.1-0.5 次版本 | 1-2 个新文档 | 新功能领域探索 |
| **交付 Cycle** | 1-3 周 | 0.1-0.3 次版本 | 更新已有文档 | 已有方向的持续交付 |
| **精化 Cycle** | 1-2 周 | 0.1 次版本 | 文档完善 | 质量提升和 Bug 修复 |
| **运营期 (Gap)** | 1-12 个月 | 维护版本 | 无需新增 | 稳定运行，积累痛点 |

**标准 Cycle 模板** (以 4 周为例)：

```
Week 1: Echo — 需求发现与验证
  Day 1-2: 沉浸在用户场景中 (实际使用、观察、记录)
  Day 3-4: 整理问题列表，定义 1 个可演示的 outcome
  Day 5:   确定本次 Cycle 的范围边界

Week 2-3: Delta — 快速原型与迭代
  Day 6-7: 搭建最简原型 (hardcoded 都可以)
  Day 8-9: 用户试用，收集即时反馈
  Day 10-14: 基于反馈迭代 2-3 轮
  Day 15:   确定方案可行后，代码不要过度打磨

Week 4: Product — 抽象与文档化
  Day 16-18: 识别可抽象的模式，写通用模块
  Day 19-20: 更新 SDD 相关章节 + 技术文档
  Day 21:   发布版本，记录 CHANGELOG
```

### 1.4 团队规模与角色分配

```
┌─────────────────────────────────────────────────────────────────┐
│                    团队规模 → 角色分配策略                        │
├──────────┬──────────────┬──────────────┬─────────────────────────┤
│ 1 人     │ 2-3 人        │ 4-6 人        │ 7+ 人                   │
├──────────┼──────────────┼──────────────┼─────────────────────────┤
│ 一人轮转  │ Echo+Product  │ Echo 1-2人   │ 可按照纯 FDE 模型       │
│ 三角色    │ 固定1人       │ Delta 2-3人   │ 一人一角色               │
│           │ Delta 1-2人   │ Product 1人   │ Echo 嵌入式             │
│           │               │               │ Delta 快速原型团队      │
│           │               │               │ Product 总部工程团队    │
│           │               │               │                         │
│ MRRC 即为 │ 角色切换成本   │ Delta 可并行   │ 需要明确的角色边界       │
│ 此模式    │ 需重点关注     │ 交付效率提升   │ 和沟通机制               │
└──────────┴──────────────┴──────────────┴─────────────────────────┘
```

**1 人团队的核心挑战与应对**：

| 挑战 | MRRC 应对方法 | 效果 |
|------|-------------|------|
| 角色切换上下文丢失 | 用文档做"状态交接"：Echo 输出的问题列表 → Delta 的 todo list → Product 的抽象清单 | 文档即状态机 |
| 看不到自己的盲区 | Gap Period (运营期) 是最长的 Echo 阶段 — 让时间暴露问题 | 等待 14 个月后发现 V4.0 需要做什么 |
| 缺乏代码审查 | 每个 Cycle 的 Product 阶段自我审视：这段代码能服务下 5 个类似需求吗？ | 抽象意识 |
| 动力枯竭 | Demo 驱动：每次交付都有可演示的 outcome，获得即时反馈 | 用户渴望是最强动力 |

## 第二部分：Echo — 需求发现实战指南

### 2.1 Echo 阶段的目标

**不是"收集需求"，是"定义有价值的 demo"。**

```
┌─────────────────────────────────────────────────────────────────┐
│                     Echo 工作流                                  │
│                                                                 │
│  沉浸观察 ──→ 问题聚类 ──→ Demo 定义 ──→ Scope 锁定             │
│                                                                 │
│  输出:                                                          │
│  1. 问题陈述 (1 页)                                              │
│  2. Demo 定义 (用户将看到什么、感受到什么)                        │
│  3. 本次 Cycle 的范围边界 (明确"不做"什么)                       │
│  4. 更新 SDD: Ch4 系统上下文, Ch5 NFR, Ch6 用例                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 Echo 实战方法

#### 方法一：运营期沉浸 (MRRC 最有效的方法)

MRRC 的 Gap Period (14 个月) 积累了以下痛点：

| 积累的痛点 | 来源 | 最终驱动 |
|-----------|------|---------|
| TX→RX 切换 2-3 秒延迟 | 长时间日常通联 | Cycle 4: 延迟优化 |
| 无功率显示，盲目发射 | 每次发射时的困惑 | Cycle 5: ATR-1000 |
| 背景噪声大，听久了累 | 长时间收听 | Cycle 6: WDSP |
| UI 陈旧，不好用 | 每天操作的不便 | Cycle 8: UI 现代化 |

**操作建议**：
- 自己的项目：**定期用 1-2 小时像新用户一样完整操作一遍**，记录每一个"不爽"
- 客户的项目：**驻场至少 1 周**，观察用户的日常工作流，不打断、不提问，只记录
- 记录格式：`场景 → 操作 → 期望 → 实际 → 情绪` (不仅是事实，还有情绪)

#### 方法二：Demo 定义模板

MRRC 每个成功 Cycle 都始于一个清晰的 Demo 定义：

```markdown
## Demo 定义: [简短标题]

### 目标场景
用户在什么情况下使用？(具体到人物、时间、环境)

### 当前体验
用户现在的操作流程是什么？有什么痛点？

### 期望体验 (Demo 目标)
演示结束时，用户将看到/感受到什么？

### Demo 脚本
1. 打开 XX 页面
2. 点击 XX 按钮
3. 看到 XX 效果
4. 用户的预期反应: "XX！" (一句惊叹)

### 验证标准
- 技术指标: [如: PTT 切换 <100ms]
- 用户反馈: [如: "终于不用等 3 秒了！"]
- 边界条件: [如: 在 4G 网络下也能流畅]

### 不做的事 (Scope 边界)
- 不处理 XX 场景
- 不优化 XX 指标
- 不做 XX 功能
```

**MRRC 案例**：

| Demo 定义 | 创造的渴望 | Cycle |
|-----------|-----------|-------|
| "按下 PTT 立即发射，松开立即接收" | "终于不用等 3 秒了！" | Cycle 4 |
| "嘈杂环境中开启 NR2，SSB 听起来像 FM" | "简直不敢相信这是短波！" | Cycle 6 |
| "一眼就看到功率和驻波，跟电台仪表一样快" | "太直观了！" | Cycle 5 |
| "比原生 App 还好看的界面" | "这是网页？不可能吧" | Cycle 8 |
| "声音温暖有温度，像坐在对面聊天" | "这才是通联该有的声音" | Cycle 9 |

#### 方法三：问题聚类

当积累足够多痛点后，用"频率 × 情绪"矩阵聚类：

```
高频 + 强烈负面情绪 = 优先级最高 (立即做下一个 Cycle)
高频 + 轻度不适     = 优先级中 (排队)
低频 + 强烈负面情绪 = 特殊性需求 (评估通用性)
低频 + 轻度不适     = 不处理或低优先级
```

**MRRC 案例**：PTT 延迟是"高频 + 强烈负面情绪"，所以 V4.0 是爆发期第一个 Cycle。

### 2.3 Echo → Delta 切换清单

在从 Echo 切换到 Delta 前，确认：

- [ ] 问题陈述已写成一句话 ("用户做 XX 时，因为 YY，导致 ZZ")
- [ ] Demo 定义已完成 (含 Demo 脚本)
- [ ] Scope 边界已明确 (本次不做的事)
- [ ] 至少有一个技术验证点 (最不确定的部分要先验证)
- [ ] SDD Ch4 (系统上下文) / Ch6 (用例) 已更新或确认无需更新

## 第三部分：Delta — 快速原型交付实战指南

### 3.1 Delta 阶段的目标

**不是"写出好代码"，是"以最短时间验证 Demo 可行"。**

Delta 阶段的代码是**消耗品**，不是**资产**。它在 Product 阶段会被抽象甚至重写，接受这一点。

```
┌─────────────────────────────────────────────────────────────────┐
│                     Delta 工作流                                  │
│                                                                 │
│  最简原型 ──→ 用户试用 ──→ 快速迭代 ──→ 方案固化                  │
│  (1-2天)     (即时)      (2-3轮)      (不追求完美)                │
│                                                                 │
│  输出:                                                          │
│  1. 可演示的功能 (不一定是最终产品)                                │
│  2. 验证过的技术方案                                             │
│  3. 已知的问题和改进点                                           │
│  4. 更新 SDD: Ch7 领域模型, Ch11 组件模型                        │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 Delta 实战方法

#### 方法一：最简原型策略

MRRC 的 Delta 阶段有几个经典的最简原型案例：

| 需求 | 最简原型 | 验证了什么 |
|------|---------|-----------|
| PTT 预热帧 | 硬编码 50ms 预热帧数组，直接 WebSocket 发送 | 预热能解决延迟问题 |
| ATR-1000 显示 | 在 Python 中直接 print 设备返回的原始数据 | 协议虽与手册不同，但数据可取 |
| NR2 降噪 | 直接复制编译好的 libwdsp.dylib，命令行调用 | 降噪效果明显，值得集成 |
| RagChew 音频链 | 在浏览器 Console 中手动创建 BiquadFilter 节点串联 | 音频链在浏览器端可行 |

**操作建议**：
- **第 1 个原型，只验证最不确定的部分**。其他全部 hardcode。
- **不写测试、不写文档、不处理异常**。反正后面要重写。
- **用 console.log / print / 硬编码路径**。不要在这个阶段搭工程框架。

#### 方法二：多轮迭代策略 — ATR-1000 案例

MRRC 的 Cycle 5 (ATR-1000) 是最典型的 Delta 迭代过程：

```
迭代 1 (V4.3.0): 在主进程中直接连接设备
    → 验证: 数据能取到，但会导致主进程假死
    → 学到: 设备连接必须独立进程

迭代 2 (V4.3.1~V4.3.8): 移到独立线程，SYNC 定期拉数据
    → 验证: 数据能稳定获取
    → 学到: sync 频率需要动态调整，频繁请求会压垮设备

迭代 3 (V4.4.0~V4.4.9): 300ms 高速轮询
    → 验证: 延迟确实大幅降低
    → 学到: 太快了，设备扛不住（"设备因频繁请求挂起"）

迭代 4 (V4.5.0~V4.5.11): 添加节流保护
    → 验证: 设备终于稳定了
    → 学到: 动态策略比固定间隔好得多

迭代 5 (V4.5.12~V4.5.18): 独立代理进程 + 动态轮询
    → 方案固化 → 进入 Product 抽象阶段
```

**迭代模式总结**：

| 迭代轮次 | 特征 | 目标 |
|---------|------|------|
| 第 1 轮 | 能跑就行的 hack | 证明可行性 |
| 第 2 轮 | 修复最大问题 | 找到真正的限制因素 |
| 第 3 轮 | 优化关键参数 | 找到最优参数空间 |
| 第 4 轮 | 加保护机制 | 保证鲁棒性 |
| 第 5 轮 | 方案固化 | 准备抽象 |

**当 5 轮还不够时**：如果迭代超过 5 轮还没稳定，可能是技术方案选错了。ATR-1000 在第 5 轮找到正确方案（独立代理），如果当时没有找到，就应该考虑换方案（比如换设备、换协议）。

#### 方法三：协议逆向工程 — 当文档是错的

ATR-1000 的协议手册与实际行为不一致，这是硬件集成中常见的情况。MRRC 的处理方法：

```
步骤 1: 用最简代码发送已知命令，打印原始 hex 响应
   raw_data = sock.recv(1024)
   print(' '.join(f'{b:02x}' for b in raw_data))

步骤 2: 对照手册逐字节分析，标记不一致的位置
   # 手册说: data[4] = SW (0=CL, 1=LC)
   # 实测:   data[3] = SW (0=LC, 1=CL)  ← 位置和映射都反了

步骤 3: 修改假设 → 验证 → 再修改 → 再验证
   直到所有字段都有稳定解释

步骤 4: 把修正后的协议写成文档 (反过来补充手册)
```

**操作建议**：遇到硬件/协议问题，不要花超过 2 小时看手册。直接发包、截数据、hex dump、猜规律。手册是参考，实测是真理。

### 3.3 Delta → Product 切换清单

- [ ] Demo 已验证可行 (用户在真实场景下认可效果)
- [ ] 核心技术方案已稳定 (不需要再大改架构)
- [ ] 已知的主要坑已踩过并记录
- [ ] 代码中有明显可复用的模式 (同一段逻辑出现了 2+ 次)
- [ ] 没有严重的性能或安全问题
- [ ] 没有遗留的 "TODO: fix later" 会影响后续使用

## 第四部分：Product — 抽象泛化实战指南

### 4.1 Product 阶段的目标

**不是"把代码写漂亮"，是"让下一次遇到类似问题时，复用成本接近于 0"。**

Product 阶段的产出是**资产**，不是**消耗品**。

```
┌─────────────────────────────────────────────────────────────────┐
│                     Product 工作流                                │
│                                                                 │
│  识别模式 ──→ 抽取通用层 ──→ 写文档/测试 ──→ 版本发布             │
│                                                                 │
│  输出:                                                          │
│  1. 通用模块/库 (可服务后续场景)                                  │
│  2. 架构决策记录 (ADR)                                           │
│  3. 技术文档 (面向开发者和运维者)                                  │
│  4. 版本更新 (CHANGELOG + 版本号)                                │
│  5. 更新 SDD: Ch1, Ch3, Ch8, Ch9, Ch14                           │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 Product 实战方法

#### 方法一："碎石路 → 高速公路" 抽象模式

这是 FDE 最核心的抽象模式。从 MRRC 中提取的完整抽象方法：

```
第一步：识别可抽象的信号
  - 这段代码在 Delta 阶段写了多少份拷贝？
  - 如果下个月要支持第 3 个类似场景，今天需要改多少行？
  - 这个模式是 MRRC 特有的还是同类项目都会遇到？

第二步：确定抽象边界
  - 抽象层应该提供什么接口？(窄接口 > 宽接口)
  - 哪些参数应该可配置，哪些应该固定？
  - 默认行为应该是什么？(对 80% 场景开箱即用)

第三步：实现抽象层
  - 先写接口 (interface / abstract class)，后写实现
  - 把 Delta 阶段的代码改为调用抽象层
  - 验证：Delta 阶段的功能行为不变

第四步：写文档
  - 接口文档：有什么方法，怎么调用
  - 配置文档：有哪些参数，默认值是什么
  - 扩展文档：如果要支持新场景，应该改哪里
```

**MRRC 完整案例**：

| 碎石路 (Delta 产物) | 抽象过程 | 高速公路 (Product 产物) |
|-------------------|---------|----------------------|
| macOS 硬编码的音频设备名 | Step 1: 所有平台都有音频设备 (共性) → Step 2: 提供 `enumerate_audio_devices()` + 平台检测 → Step 3: `audio_interface.py` 统一接口 → Step 4: 配置文档 + 平台适配指南 | `audio_interface.py` |
| 直接 sleep(2) 等待电台响应 | 识别: 等电台是通用需求 → 抽象: 可配置的预热帧数量/间隔 → 实现: 预热帧 + 计数超时 → 文档: PTT 可靠性框架 | PTT 预热帧机制 |
| ATR-1000 在 MRRC 中直接连接 | 识别: 硬件代理是通用模式 → 抽象: Unix Socket 隔离的独立代理 → 实现: 动态轮询三态 (idle/active/tx) → 文档: 独立代理架构 | 独立代理架构 |
| 手动 cp libwdsp.dylib 到系统目录 | 识别: 库路径问题 → 抽象: 多路径自动检测 → 实现: 系统 → Homebrew → 项目目录 → 文档: 搜索顺序说明 | 库路径自动检测 |
| deploy.sh 中硬编码服务器 IP | 识别: 部署自动化 → 抽象: inventory 驱动的声明式部署 → 实现: Ansible playbook → 文档: inventory 配置指南 | Ansible 部署自动化 |

## 第五部分：完整 Case Study — 一个 Cycle 的全过程

以 MRRC 的 **Cycle 5 (ATR-1000 天调集成)** 为例，展示一个完整 FDE Cycle 从头到尾的过程。

### 5.1 Echo 阶段 (约 2 天)

**Day 1: 发现需求**
- 背景：用户通联时，经常不知道天线匹配好不好，发射功率够不够
- 问题：现有系统没有任何功率/SWR 反馈，只能"盲发"
- 机会：用户刚买了一个 ATR-1000 天调设备，支持 WebSocket 远程数据获取

**Day 2: 定义 Demo**
```
Demo 定义: 在手机屏幕上看到实时功率变化 (像本地电台仪表一样)

目标场景: 用户按下 PTT 后，看到手机屏幕上功率条跳动
期望体验: "一眼就知道天线效率好不好"
Demo 脚本:
  1. 打开移动端页面
  2. 看到功率/SWR 显示区域
  3. 按下 PTT 开始发射
  4. 功率条实时跳动 (延迟 <200ms)
  5. 松开 PTT，功率归零

Scope 边界:
  - 本次只做功率/SWR 显示，不做智能学习，不做快速调谐
  - 只支持 ATR-1000 一种型号
  - 不做历史数据图表

SDD 更新:
  - Ch4 系统上下文: 新增 ATR-1000 外部实体
  - Ch6 用例模型: 新增 UC-004 Antenna Tuner Control
  - Ch5 NFR: 新增功率显示延迟 <200ms 要求
```

### 5.2 Delta 阶段 (约 10 天, 5 次迭代)

**迭代 1 (Day 3-4): 最简连接验证**
- 实现：在 MRRC 主进程中直接 connect ATR-1000 的 WebSocket
- 结果：能收到数据！但...发射时会阻塞主进程，TX→RX 切换延迟从 100ms 变成 2 秒
- 学到：不能挂在主进程上

**迭代 2 (Day 5-6): 独立线程**
- 实现：开一个独立 Python 线程跑 WebSocket 连接
- 结果：不阻塞主进程了。但数据更新太慢，设置了 300ms 高频 SYNC
- 学到：线程安全问题 — 跨线程写 Tornado WebSocket 需要 IOLoop.add_callback

**迭代 3 (Day 7-8): 加快轮询**
- 实现：sync 间隔从 1s 缩短到 300ms
- 结果：数据更新快了，但 ATR-1000 设备有时会"假死"（不响应请求）
- 学到：设备有处理能力上限，不能无限增压

**迭代 4 (Day 9-10): 节流保护**
- 实现：加入双重时间戳保护，最小 sync 间隔锁定 500ms
- 结果：设备稳定了，但用户反馈"TUNE 模式更新快，PTT 发射时更新慢"
- 学到：不同模式下设备数据推送行为不同 — PTT 时设备自身在采样，推送频率更低

**迭代 5 (Day 11-12): 独立代理 + 动态策略**
- 实现：`atr1000_proxy.py` 作为独立进程运行，通过 Unix Socket 与 MRRC 通信
- 策略：空闲 15s / 活跃 5s / TX 0.5s 三态动态轮询
- 结果：设备稳定，数据实时，架构解耦。方案固化。
- 学到：这是正确的架构 — 所有硬件都该用独立代理

### 5.3 Product 阶段 (约 4 天)

**Day 13-14: 抽象**
- 识别模式：独立代理 + Unix Socket + 动态轮询 → 可以抽象为通用硬件代理框架
- ADR 记录：AD-008: ATR-1000 Proxy Architecture
- 产出：`atr1000_proxy.py` 最终架构文档

**Day 15-16: 文档与发布**
- Layer 3 文档：
  - `docs/ATR1000_Tuner_Auto_Learning.md` — 完整技术文档
  - `docs/ATR1000_Display_Issue_Analysis.md` — 问题分析文档
- Layer 1/2 更新：
  - DESIGN/CONTEXT.md: 新增 ATR-1000 数据流
  - SDD/09: 更新架构图包含 ATR 代理
  - SDD/11: 新增 ATRProxyClient 组件
- 版本发布：V4.5.17

**后续扩展 (Land & Expand)**：
Cycle 5 之后，ATR-1000 的能力持续扩展：
- V4.5.16: 智能学习 (自动记录频率→参数映射)
- V4.5.17: 快速调谐 (切频自动应用已学参数)
- V4.5.18: REST API Server (外部软件可调用)
- V4.9.3: FrequencySyncThread (JTDX/flrig 联动)

每个扩展都是一个新的 mini-Cycle，但都建立在 Cycle 5 的产品杠杆上 — **部署新功能需要的代码量越来越少**。

## 第六部分：反模式与陷阱

### 6.1 Echo 阶段反模式

| 反模式 | 表现 | 后果 | 正确做法 |
|--------|------|------|---------|
| **需求膨胀** | Echo 阶段定义了 10 个 Demo，Scope 太大 | Delta 无法在 2-3 周内交付任何一个 | 每次只选 1 个 Demo。完成后再做下一个。 |
| **假 Echo** | 没去现场，凭想象写需求 | 做出来的功能没人用 | 至少在实际场景中使用 1 周再定义需求 |
| **过度分析** | 写了 50 页需求文档还没开始写代码 | 进入 Delta 时已经过了最佳窗口期 | Echo 产出 1 页纸就够了。够开始 Delta 就行。 |
| **跳过了运营期** | 刚做完就做下一个，没给真实使用留时间 | 永远不知道真正的痛点 | 每个大版本后至少留 2-4 周的运营观察期 |

### 6.2 Delta 阶段反模式

| 反模式 | 表现 | 后果 | 正确做法 |
|--------|------|------|---------|
| **过早优化** | 第 1 个原型就写完整错误处理、日志、配置 | 2 天后发现方案错了，全部白写 | 第 1 个原型只验证假设，后续迭代再加工程化 |
| **无限迭代** | 迭代了 10 轮还在调参 | 永远无法进入 Product 阶段 | 5 轮迭代后还没稳定 → 考虑换方案 |
| **过度工程化** | Delta 阶段就开始写抽象层 | 抽象层在 3 轮迭代后全部推倒 | Delta 阶段不写抽象层，直接硬编码。Product 阶段才抽象 |
| **没有 Demo** | 一直在写代码，但从不给用户演示 | 方向可能已经完全偏离 | 每 2-3 天给用户演示一次当前状态 |

### 6.3 Product 阶段反模式

| 反模式 | 表现 | 后果 | 正确做法 |
|--------|------|------|---------|
| **过早抽象** | 第 1 个场景还没跑通就写通用框架 | 框架不匹配任何实际场景 | 至少 2 个类似场景成功后再抽象 |
| **过度抽象** | 抽象层兼容了 20 种未来可能但实际不会发生的场景 | 接口复杂，无人会用 | YAGNI: You Ain't Gonna Need It |
| **跳过抽象** | Delta 的代码直接发布，不做 Product 化 | 下个场景要重写全部代码 | 每个 Cycle 必须留至少 20% 时间给 Product |
| **只写代码不写文档** | 高速公路修好了但没有路牌 | 下个人 (包括 3 个月后的自己) 完全不知道怎么用 | 每个抽象模块至少需要：接口文档 + 一个使用示例 |

### 6.4 角色切换陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| **角色混淆** | 在 Delta 阶段还在纠结需求定义 | 严格按 Echo→Delta→Product→Echo 顺序，不跳步 |
| **切换不彻底** | Echo 输出模糊，Delta 无法执行 | Echo 必须有具体输出物 (问题陈述 1 页 + Demo 定义) |
| **Product 偷懒** | Delta 完成后觉得 "够好了" 就不做抽象 | 强制在时间表中给 Product 留 20-25% 时间 |

## 第七部分：度量和 KPI

### 7.1 FDE 健康度指标

| 指标 | 健康值 | 警戒值 | 测量方法 |
|------|--------|--------|---------|
| **Cycle 周期** | 2-4 周/Cycle | >6 周 | 每个版本的时间跨度 |
| **Demo 验证率** | >80% Demo 在 Delta 结束时可演示 | <50% | 回顾时统计 |
| **抽象产出率** | 每个 Cycle >=1 个通用模块 | 连续 2 个 Cycle 无抽象产出 | 回顾时统计 |
| **产品杠杆增速** | 每个 Cycle +1 | 连续 2 个 Cycle 不变 | 见 7.2 |
| **文档完成率** | 每个 Cycle 更新 SDD >=3 章 | 连续 Cycle 不更新 SDD | 回顾时统计 |
| **迭代收敛速度** | 3-5 轮迭代到方案稳定 | >8 轮 | 回顾时统计 |

### 7.2 产品杠杆测量

这是 FDE 最核心的量度：**同样的成果，后续交付需要的代码量是否减少？**

```
产品杠杆 = 当前可以服务的场景数 / 新增一个场景需要的边际代码量

示例 (MRRC 的 audio_interface.py):
  - Delta 阶段: macOS 硬编码 1 个平台, 代码 ~200 行
  - Product 阶段: audio_interface.py 支持 3 个平台, 代码 ~300 行
  - 产品杠杆 = 3 / (300/3 - 200) ≈ 有了抽象后，新平台只需要配置，不需要写代码
  - 杠杆值 = ∞ (新增场景代码量为 0 或接近 0)
```

### 7.3 SDD 健康度指标

| 指标 | 健康值 | 说明 |
|------|--------|------|
| SDD 版本与代码版本一致 | 100% | CHANGELOG 和 SDD/14 应完全对齐 |
| ADR 覆盖率 | 每个重大架构决策有 ADR | 选择技术栈、架构模式、关键依赖时必写 |
| NFR 有验证方法 | 100% | 每个非功能需求都有可执行的验证方法 |
| 用例与实际功能一致 | >90% | 做过的功能都有用例记录 |

```
┌──────────────────────────────────────────────────────────────────┐
│                    FDE Cycle (2-4 周)                             │
│                                                                  │
│   时间:  ┌──── Week 1 ────┬─── Week 2-3 ───┬─── Week 4 ────┐   │
│   阶段:  │     Echo       │     Delta      │    Product     │   │
│   焦点:  │  "需要什么"     │  "怎么做出来"   │  "如何复用"    │   │
│   产出:  │  问题+Demo     │  原型+验证     │  模块+文档     │   │
│   SDD:   │  Ch4,5,6       │  Ch7,11,12    │  Ch1,3,8,9,14 │   │
│                                                                  │
│   原则:  │ Demo 定义驱动   │ 迭代≤5轮      │ 至少1个ADR     │   │
│          │ Scope 必须锁    │ 每2天Demo一次  │ 至少3章SDD     │   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 关键原则速查

1. **Demo 驱动，不是需求驱动** — 需求文档会过时，Demo 脚本不会
2. **先碎石路，后高速公路** — 不要跳步，先验证再抽象
3. **运营期是 Echo** — 持续使用是最好的需求发现方式
4. **Delta 的代码是消耗品** — 不要投入感情，接受它会重写
5. **Product 的文档是资产** — 让 6 个月后的自己能快速上手
6. **每个 Cycle 必须产出产品杠杆** — 不然就是在原地踏步
7. **5 轮迭代收不住就换方案** — 不要无限打磨
8. **1 人团队的最大敌人是角色混淆** — 用文档做状态交接

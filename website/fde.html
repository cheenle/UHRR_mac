Methodology

What is FDE?

Forward Deployed Engineering — Palantir's outcome-driven product discovery and delivery model

FDE = Echo + Delta + Product
Demand Discovery → Rapid Prototyping → Abstraction & Generalization

Echo

"What does the user really need?"
  • Embedded in user scenarios
  • Define valuable demos
  • Manage client relationships
  • Problem discovery & scoping
  • ~20% of cycle time

Delta

"What's the fastest thing that works?"
  • Rapid prototyping
  • Deliver within timeline
  • Code can be rough but functional
  • Iterate based on feedback
  • ~55% of cycle time

Product

"How to serve the next 10 customers?"
  • Abstract field solutions
  • Build reusable modules
  • Write architecture docs
  • Turn gravel roads into highways
  • ~25% of cycle time

FDE is not consulting. Consulting is a one-time service; FDE is a product exploration process. FDE paves "gravel roads" in the field, while headquarters builds "highways". The unit cost of delivering outcomes decreases over time — margins turn from negative to positive.

Core Patterns

Three Signature FDE Patterns

Distilled from 12 cycles of MRRC development

🛤️ → 🛣️

Gravel Road → Highway

The core FDE abstraction pattern. Delta-phase hacks ("gravel roads") are refined in the Product phase into reusable, generic modules ("highways") that serve the next 5-10 customers with near-zero marginal cost.

Gravel: ALSA Linux-only audio ↓ Abstract Highway: PyAudio cross-platform unified interface Gravel: Direct ATR-1000 connection crashing main process ↓ Abstract (5 iterations) Highway: Standalone proxy + dynamic polling (15s/5s/0.5s)
🎯 → 🌐

Land & Expand

Enter with one concrete problem (Land), discover adjacent problems from real usage, expand scope organically. Each expansion builds on prior leverage.

Land: Power/SWR real-time display Expand: Smart learning → Quick tune → REST API → External software integration → 3rd-party frequency sync Land: Basic audio streaming Expand: WDSP NR → TX EQ → RagChew → Full TX/RX audio processing chain
✨

Demo-Driven Development

Define a concrete demo at the Echo stage. Every iteration is validated by showing the user a working demo. The demo creates desire — the strongest motivation to ship.

Demo: "Press PTT, transmit instantly" → Desire: "No more 3-second wait!" → Cycle 4: PTT pre-warm frames Demo: "NR2 on, SSB sounds like FM" → Desire: "I can't believe this is HF!" → Cycle 6: WDSP NR2 integration Demo: "Warmer, natural voice on TX" → Desire: "Sounds like sitting across the table" → Cycle 9: RagChew audio chain

Metrics

Product Leverage Evolution

Core FDE metric: does the marginal cost of delivering the same outcome decrease over time?

What is "Product Leverage"?

Product leverage = the count of accumulated reusable assets. In each FDE cycle's Product phase, the Delta-phase throwaway code ("gravel road") is refined into a reusable, generic module ("highway"). This module becomes a permanent asset for the team. Every subsequent cycle can build directly on top of these assets instead of reinventing the wheel.

Analogy: Building a skyscraper. Cycle 1 lays the foundation (WebSocket channel). Cycle 2 installs plumbing and wiring (audio framework). Cycle 6 builds the elevator (DSP pipeline). By Cycle 12, you're just adding one room on the top floor (FT8 integration) — the foundation, utilities, and elevators are already there. You write very little code, but deliver enormous functional value. That's leverage.

How it's measured: Each cycle's Product phase produces one (or more) independently reusable modules, adding +1 (or +N) to the leverage count. Starting from 0 (a raw fork with zero proprietary assets), reaching 16 reusable modules by V5.6.5.

Evidence: cycle duration shrinks as leverage grows

If leverage is real, the most direct evidence is progressively shorter development cycles. The table below groups cycles by leverage tier and compares average duration:

Leverage TierCyclesAvg DurationReusable Assets Available
0–3Cycle 1–3 (V1.0–V3.0)~4 weeksNearly zero. WebSocket, audio, and mobile all built from scratch.
4–7Cycle 4–6 (V4.0–V4.9)~1–3 weeksAudio framework, PTT framework, mobile framework ready. New features plug in directly.
8–10Cycle 7–9 (V5.0–V5.1)~1–1.5 weeksDSP pipeline, ATR proxy, Ansible deploy all in place. New features are plug-and-play.
11–13Cycle 10–12 (V5.2–V5.4)~0.5–1 weekDesign system, audio presets, network monitoring all baked in. Each new feature needs minimal incremental code.
14–16Cycle 13–15 (V5.5–V5.6.5)~1–2 weeksMemoryChannelManager, LearningBuffer, EFHW knowledge base. Frontend optimization, ATR stability, antenna research.

Conclusion: Cycle 1 took 4 weeks to build the foundation. Cycle 15 took 1 week to integrate EFHW documentation — both "delivering a complete feature," but the latter required only 1/4 the actual effort. The developer didn't get faster. The 16 reusable assets did the heavy lifting. That's product leverage in action: every delivery reduces the cost of the next.

FDE + IBM TeamSD (SDD)

IBM TeamSD v2.3.2's 14-chapter SDD provides the structural framework for FDE cycles. FDE drives the content; SDD provides the format.

DeltaCh7 Subject Area Model, Ch11 Component Model, Ch10 Service Model, Ch12 Operational ModelMid-to-late cycle

Core principle: Don't let documentation drive development — let development drive documentation. Update SDD chapters after each FDE phase, not before.

Best Practices

8 FDE Principles from MRRC

What worked, distilled into actionable principles

1

Start from the field, not from the product

Every cycle is driven by user scenarios and real problems — never "I want to add this feature." The 14-month Gap Period proved that real usage is the best Echo.

2

Demo-driven validation

Every delivery validates user desire through an actual demo, not paper planning. PTT demo, NR2 demo, RagChew demo — each created genuine user excitement.

3

Timely abstraction

Each cycle ends with a Product-phase abstraction. Don't abstract too early (before 2+ similar use cases), don't skip it entirely. The Delta code is consumable; Product output is an asset.

4

Documentation as output

55+ design and technical docs, 14-chapter SDD, 6 DESIGN files — knowledge must be transferable. Your 6-months-later self is the primary audience.

5

Cross-platform from day one

V2.0 replaced ALSA with PyAudio from the start, avoiding Linux lock-in. Cross-platform is not a feature — it's architecture.

6

Operations is Echo

The 14-month "quiet period" accumulated real pain points that drove explosive innovation (6 minor versions in 12 weeks). Don't skip the soak time.

7

5 iterations, then pivot

If a solution hasn't stabilized after 5 iterations, the technical approach is probably wrong. ATR-1000 found the right architecture on iteration 5. Don't polish indefinitely.

8

Documentation as state machine (solo dev)

For 1-person teams, use documentation to hand off state between role switches: Echo outputs → Delta todo list → Product abstraction checklist.

Anti-Patterns

Common FDE Pitfalls

What to watch out for at each phase

Requirement Bloat (Echo)

Defined 10 demos with an enormous scope

Pick 1 demo per cycle. Ship it. Then do the next.

Fake Echo

Imagined requirements without field presence

Spend at least 1 week in the actual usage scenario before defining requirements.

Premature Optimization (Delta)

Full error handling + logging + config in prototype #1

Prototype #1 only validates the riskiest assumption. Engineering comes later.

Infinite Iteration (Delta)

10 iterations still tuning parameters

If not stable after 5 iterations, consider switching the technical approach entirely.

Premature Abstraction (Product)

Writing a generic framework before scenario #1 is validated

Wait until at least 2 similar scenarios succeed before abstracting.

Skipping Product Phase

Delta code shipped directly without generalization

Reserve at least 20% of every cycle for the Product phase. Every cycle.

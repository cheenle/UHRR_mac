方法论

什么是 FDE？

Forward Deployed Engineering — Palantir 以结果为导向的产品发现与交付模型

FDE = Echo + Delta + Product
需求发现 → 快速原型 → 抽象与泛化

Echo（回响）

"用户真正需要什么？"
  • 嵌入用户真实场景
  • 定义有价值的 Demo
  • 管理客户关系
  • 问题发现与范围界定
  • 约占 Cycle 时间 20%

Delta（增量）

"最快能跑通的是什么？"
  • 快速开发原型
  • 在规定时间内交付
  • 代码可以粗糙但必须可用
  • 根据反馈持续迭代
  • 约占 Cycle 时间 55%

Product（产品化）

"如何服务下 10 个客户？"
  • 将一线方案抽象沉淀
  • 构建可复用的模块
  • 编写架构文档
  • 把碎石路铺成高速公路
  • 约占 Cycle 时间 25%

FDE 不是咨询。 咨询是一次性服务，FDE 是产品发现过程。 FDE 在一线铺设"碎石路"，总部在此基础上修建"高速公路"。 交付同一成果的单位成本随时间递减——边际利润率从负转正。

核心模式

三大 FDE 标志性模式

从 MRRC 的 12 个 Cycle 中提炼而来

🛤️ → 🛣️

碎石路 → 高速公路

FDE 的核心抽象模式。Delta 阶段的临时代码("碎石路")在 Product 阶段被精炼为可复用的通用模块("高速公路")，以接近零的边际成本服务下 5-10 个客户。

碎石路：ALSA Linux 独占音频 ↓ 抽象 高速公路：PyAudio 跨平台统一接口 碎石路：ATR-1000 直接连接导致主进程崩溃 ↓ 抽象 (5次迭代) 高速公路：独立代理 + 动态轮询 (15s/5s/0.5s)
🎯 → 🌐

登陆 & 扩张

以一个具体问题切入(Land)，从实际使用中不断发现相邻问题，有机地扩展范围。每次扩张都建立在此前积累的杠杆之上。

登陆：功率/驻波实时显示 扩张：智能学习 → 快速调谐 → REST API → 外部软件集成 → 第三方频率同步 登陆：基础音频流 扩张：WDSP NR → TX EQ → RagChew → 全链路 TX/RX 音频处理
✨

Demo 驱动开发

在 Echo 阶段就定义好具体的 Demo。每次迭代都通过给用户展示可工作的 Demo 来验证。Demo 创造渴望——这是最强的交付动力。

Demo："按下 PTT，瞬间发射" → 渴望："不用再等3秒了！" → Cycle 4：PTT 预加热帧 Demo："NR2 一开，SSB 听起来像 FM" → 渴望："这竟然是短波？！" → Cycle 6：WDSP NR2 集成 Demo："TX 声音更温暖自然" → 渴望："就像坐在对面说话" → Cycle 9：RagChew 音频链路

度量指标

产品杠杆演进

FDE 核心指标：交付同一成果的边际成本是否随时间递减？

什么是"产品杠杆"？

产品杠杆 = 累积的可复用资产数量。 每个 FDE Cycle 的 Product 阶段，都会把 Delta 阶段的临时代码（"碎石路"）提炼成一个可复用的通用模块（"高速公路"）。 这个模块沉淀下来，成为团队的永久资产。后续 Cycle 做任何新功能，都可以直接站在这些资产之上，不必重复造轮子。

类比：就像盖楼。Cycle 1 你还在打地基（WebSocket 通道），Cycle 2 你搭了水电管线（音频框架），Cycle 6 你装好了电梯（DSP 管线）。 到了 Cycle 12，你只需要在顶楼加一个房间（FT8 集成）—— 地基、管线、电梯全都是现成的。你实际写的代码量很少，但交付的功能价值却很大。这就是"杠杆"。

衡量方式：每个 Cycle 的 Product 阶段产出一个（或多个）可独立复用的模块，杠杆数 +1（或 +N）。 杠杆从 0 开始（只有一份原始 fork，没有任何自有资产），到 V5.6.5 累计 16 个可复用模块。

柱高 = 累积可复用资产数量。每个 Cycle 都在上一 Cycle 的基础上"叠罗汉"，而非从零开始。

证据：Cycle 耗时随杠杆增长而递减

如果杠杆真的在起作用，最直接的证据就是开发周期越来越短。 下表按杠杆水平分组，对比平均 Cycle 耗时：

杠杆区间Cycle平均耗时当时可复用的资产
0–3Cycle 1–3 (V1.0–V3.0)~4 周几乎为零。WebSocket、音频、移动端全部从零构建。
4–7Cycle 4–6 (V4.0–V4.9)~1–3 周音频框架、PTT 框架、移动端框架已就绪。新功能直接复用。
8–10Cycle 7–9 (V5.0–V5.1)~1–1.5 周DSP 管线、ATR 代理、Ansible 部署全部到位。新功能即插即用。
11–13Cycle 10–12 (V5.2–V5.4)~0.5–1 周设计系统、音频预设、网络监控均已沉淀。每个新功能只需极少量增量代码。
14–16Cycle 13–15 (V5.5–V5.6.5)~1–2 周MemoryChannelManager、LearningBuffer、EFHW知识库。前端优化、ATR稳定性、天线研究。

结论：Cycle 1 花 4 周搭建基础架构，Cycle 15 花 1 周集成 EFHW 文档 —— 同样是"交付一个完整功能"，后者的实际投入仅为前者的 1/4。 这背后的原因不是开发者变快了，而是 16 个可复用资产在替他工作。这就是产品杠杆的本质：每一次交付都在降低下一次交付的成本。

方法论整合

FDE + IBM TeamSD (SDD)

IBM TeamSD v2.3.2 的 14 章 SDD 为 FDE Cycle 提供了结构化框架。FDE 驱动内容，SDD 提供格式。

FDE 阶段驱动 SDD 章节时机
Echo第4章 系统上下文、第2章 业务方向、第5章 非功能需求、第6章 用例模型、第13章 可行性分析首次 Echo + 每个 Cycle
Delta第7章 主题域模型、第11章 组件模型、第10章 服务模型、第12章 运维模型Cycle 中后期

核心原则：不要让文档驱动开发——让开发驱动文档。在每个 FDE 阶段之后更新 SDD 章节，而非之前。

最佳实践

来自 MRRC 的 8 条 FDE 原则

被验证有效的做法，提炼为可操作的原则

1

从一线出发，而非从产品出发

每个 Cycle 都由用户场景和真实问题驱动——从不"我想加一个功能"。14个月的 Gap Period 证明真实使用是最好的 Echo。

2

Demo 驱动验证

每次交付都通过实际 Demo 来验证用户渴望，而不是纸上规划。PTT Demo、NR2 Demo、RagChew Demo——每个都创造了真切的用户兴奋感。

3

适时抽象

每个 Cycle 都以 Product 阶段收尾。不要太早抽象（2个以上相似案例之前），也不要完全跳过。Delta 代码是消耗品，Product 输出才是资产。

4

文档即输出

55+份设计和技术文档、14章SDD、6份 DESIGN 文件——知识必须可转移。六个月后的你自己才是主要读者。

5

从第一天起跨平台

V2.0 从一开始就用 PyAudio 替代了 ALSA，避免了 Linux 锁定。跨平台不是功能——它是架构。

6

运维即 Echo

14个月的"静默期"积累了大量真实痛点，驱动了爆发式创新（12周内发布6个小版本）。不要跳过浸泡期。

7

5次迭代，然后转向

如果一个方案在5次迭代后仍未稳定，技术路径可能根本就是错的。ATR-1000 在第5次迭代才找到正确架构。不要无限打磨。

8

文档即状态机（单人团队）

对于1人团队，用文档在角色切换之间传递状态：Echo 输出 → Delta 待办列表 → Product 抽象检查清单。

反模式

常见 FDE 陷阱

在各阶段需要警惕的问题

需求膨胀 (Echo)

一次定义10个Demo，范围巨大

每个Cycle只选1个Demo。交付后再说下一个。

虚假 Echo

凭想象写需求，缺乏一线实际体验

在定义需求前，至少花1周时间在真实使用场景中。

过早优化 (Delta)

第一个原型就写完整错误处理+日志+配置

1号原型只需验证风险最高的假设。工程化靠后。

无限迭代 (Delta)

迭代了10次还在调参数

5次迭代还不能稳定，说明技术方案可能根本不对。

过早抽象 (Product)

场景1还没验证就开始写通用框架

至少等2个相似场景都成功后再抽象。

跳过 Product 阶段

Delta 代码直接发货，不做泛化

每个 Cycle 至少留 20% 的时间给 Product 阶段。每个 Cycle。

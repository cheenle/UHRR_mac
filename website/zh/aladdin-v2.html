定义

什么是阿拉丁 V2？

一套把 IT 老兵经验转化为可执行、可复用、可由 agent 协作的软件工程资产的方法。

阿拉丁 V2 = 老灯新灵 + FDE Cycle + SDD Spine + Agentic Harness + Product Leverage
老灯新灵 = Veteran Judgment + Vibe Coding + Field Truth
老兵定义方向和边界，新灵释放执行速度，现场事实校验两者。
阿拉丁神灯的故事里，旧灯看似普通，灯灵被封印多年；当灯被擦亮，沉睡的力量被释放。对应到软件工程：老灯是 IT 老兵长期积累但常被封存在个人经验里的判断力；新灵是 vibe coding 和 AI agent 释放出的智能执行力。阿拉丁 V2 把这种释放变成受约束、可复用的工程系统。
核心循环

三层，一个循环

七层模型坍缩成一个可实操的循环。每层只回答一个问题、只背一条约束、只换一个效果。

L1 · 定界

先画边界

谁：老兵 + 现场事实

动手前先定：做什么、不做什么、必须守住什么。

约束先写边界卡；超过 10 行说明还没想清楚。
效果把 AI 的高速猜测变成高速执行。
L2 · 提速

在界内跑快

谁：AI / agent，在边界内

Vibe coding 出速度，FDE 节奏（Echo→Delta→Product）定方向。

约束先验最大不确定性；3–5 轮收敛；每轮能 demo。
效果边界刚刚为你换来的那份速度。
L3 · 沉淀

变成资产

谁：人 + AI 一起

SDD、harness、ADR、postmortem——能活过本轮的载体。

约束每轮至少留一个可复用资产，否则不算完成。
效果下次更便宜——这就是杠杆。
边界 → 速度 → 资产 →（更省的）边界
工作流

循环的一圈

定界、提速、沉淀——然后下一次边界更省力。

L1先画边界
L2在界内跑快
L3沉淀成资产
MRRC 实践

为什么 MRRC 是验证场

真实硬件

电台、USB 声卡、ATR-1000、串口、RTL-SDR、TLS 和本地依赖让方案不能停留在想象中。

实时约束

PTT 时序、TX/RX 队列、AudioWorklet 缓冲、Opus 帧对齐和 WebSocket 调度暴露真实工程风险。

完整架构记忆

FDE 复盘、SDD 14 章、DESIGN 文档、postmortem、AGENTS 和 agentic plans 构成完整知识闭环。

本地 Skill

方法论已落地为 OpenCode skill：.opencode/skills/aladdin-v2/SKILL.md。

原则

阿拉丁 V2 规则

1

L1先写边界，再写代码。没有边界，就不放 agent。

2

L1现场事实高于观点；速度只存在于边界之内。

3

L2Delta 代码是消耗品；沉淀下来的资产才是你留住的东西。

4

L3每个 Cycle 都必须让下一个更便宜——否则它还没完成。

5

∞方法论要标注自己的射程；出了射程，自己补装备。

批判性辩证

客观一面：成熟度高度不均衡

一个标注了自己边界的方法论，才是成熟的方法论。下面是阿拉丁 V2 强在哪里，又在哪里从未被压力测试过。

阿拉丁 V2 不是一个不成熟的方法论，而是一个成熟度高度不均衡的方法论。在单人硬件项目反复锤炼的维度上（执行约束、知识沉淀、架构记忆），它达到了少见的工程密度；在该项目从未施加压力的维度上（多人判断、权威纠错、预算与安全、高频反馈），它接近空白。这些空白不是设计失败，而是尚未被探索的射程。

已被实物验证

  • 可执行 plan 带精确行号、完整代码块、py_compile 验证命令、预期输出，甚至预写 git commit。
  • SDD 骨架：14 章完整存在，不是空壳目录。
  • Memory harness：多份 postmortem 已固化进 AGENTS.md guardrail。
  • 可触发的 OpenCode skill，有真正的 phase detection，而非动机性散文。

从未被压力测试

  • 团队维度：两个老兵判断冲突时无仲裁；没有老兵时无冷启动路径。
  • 纠错回路：老兵判断是最高权威，当它本身错时没有证伪入口。
  • 成本与安全：token 失控、agent 误碰生产、供应链投毒——只字未提。
  • 反馈延迟：从未作为一等变量建模。
正确的态度不是"这个方法论有缺陷所以打折用"，而是：在它的射程内是精良的，出了射程要自己补装备——而射程边界，原作者没标，这才是唯一真正的疏漏。下面这套工具包，就是那套装备。
业务工具包

驶入新水域的五个工具

按需取用。最低门槛：只用边界卡。其余四个随团队成熟度逐步引入。

1 · 边界卡

10 行以内，在任何人（或 agent）动手前写下：目标、不做什么、必须守住什么、最容易猜错的 3 点、失败如何被发现。ROI 最高、迁移成本最低。

2 · 七类 Harness

原六类（Context / Spec / Plan / Verification / Runtime / Memory）加新增第七类 Safety/Cost harness：agent 能碰什么、预算上限、生产环境前的人工确认点。

3 · 三个硬指标

用可证伪的趋势线替代定性评分：接手探索时间、同类事故复发次数、相似功能边际成本。曲线下降，杠杆才是真的。

4 · 纠错回路

每个关键决策点插入一道轻量挑战：同伴反向论证、"如果老兵错了"压力测试、或证据来源审计。给单一判断源一个证伪入口。

5 · 失效场景标注

引入前先标注适用性：领域稳定、判断集中、反馈周期长 = 高适用。需求高频变化或合规高于现场的领域，需大幅改写 FDE 节奏。

每个工具补一个盲点

纠错回路 ↔ 单英雄空白 · 失效标注 ↔ n=1 过拟合 · 硬指标 ↔ 定性自我安慰 · Safety/Cost harness ↔ 成本与安全的沉默。
